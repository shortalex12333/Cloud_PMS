## 1. WHAT THIS MODULE IS

Cloud PMS is a yacht management system. The **Hours of Rest (HoR)** module exists to comply with MLC 2006 maritime law. The law requires that every crew member logs their rest hours daily. Minimums: 10 hours rest per 24-hour period, 77 hours per 7 days, no more than 2 rest periods per day, and the longest rest period must be at least 6 hours. Violations must be flagged, reported to the crew's Head of Department (HOD), and signed off through a chain: Crew → HOD → Captain → Fleet Manager.

The module covers: daily time entry, weekly compliance checking, violation detection, sign-off workflow, notifications, and fleet-level oversight.

### 2a. Business Summary

Three things happened this session:

1. **The integration test suite was hardened to only accept real evidence.** Previously, many tests were passing based on fake or unconditional checks — a test that always passed regardless of the actual API or database result. These were replaced with checks that demand specific JSON fields or database row values. Nine scenarios, 92 checks, all backed by real evidence.

2. **Three backend bugs were discovered and fixed** during testing. These bugs were silent — they did not crash the system but produced wrong results. One prevented monthly lock enforcement (crew could submit records into a locked month). One caused notifications to go to nobody when a crew member had a department assigned. One caused a 500 error when dismissing a warning that didn't exist.

3. **The work/rest inversion was designed and built.** The original system asked crew to input their REST hours. This was changed: crew now input their WORK hours, and the backend automatically calculates rest as the inverse (24 hours minus work). A blank day means 24 hours of rest — correct for a day off — not zero rest, which was the previous (wrong) behaviour.

## 3. FILES CHANGED

### BACKEND

#### `apps/api/handlers/hours_of_rest_handlers.py`
**Purpose:** The core business logic handler. Every HoR action — submitting a day, creating a signoff, signing, flagging violations, notifications — lives here.
**Status:** Edited (multiple fixes + inversion)

Changes made:
- **Work/rest inversion (upsert_hours_of_rest):** Handler now accepts `work_periods: [{start, end}]` instead of `rest_periods`. It derives `rest_periods` as the 24-hour complement using a `_complement()` function, then stores both. Empty work_periods = 24h rest = valid day off. Previously, missing rest_periods was a validation error; now empty work_periods is valid.
- **Validation added:** Each `work_period` must have `start` and `end` in HH:MM format. This was added after discovering FRONTEND02's old tests sent `{hours: N}` with no time data — the backend was silently storing timeless records.
- **Monthly lock fix:** The lock check was silently failing because `maybe_single()` was used on a query that could return multiple rows, causing it to return None. The None was caught and swallowed. Fixed by switching to `.execute()` and checking with `any()`.
- **Notification indentation bug:** The notification dispatch code (which alerts the HOD when a crew member submits a violating day) was accidentally inside the `else:` branch of an `if dept:` block. When a crew member had a department, the `if dept:` branch ran but the notification code was in `else:` — so notifications were never sent for crew with a department. Fixed by moving the notification code outside both branches.
- **dismiss_warning null crash:** `maybe_single()` returns Python None when zero rows are found. Accessing `.data` on None throws an AttributeError, which was silently caught and returned as DATABASE_ERROR. Fixed with an explicit null check before accessing `.data`.
- **Notification column names:** INSERT into `pms_notifications` was using wrong column names (`recipient_user_id`, `type`, `message`). Actual DB columns are `user_id`, `notification_type`, `body`. Fixed.

#### `apps/api/routes/hor_compliance_routes.py`
**Purpose:** Three GET endpoints that power the main HoR UI views — one for each role: crew (my-week), HOD (department-status), Captain (vessel-compliance).
**Status:** Edited

- `my-week` query now selects `work_periods` alongside `rest_periods` from `pms_hours_of_rest`
- Unsubmitted day slots now include `work_periods: []` so the frontend slider always has the field available

#### `apps/api/pipeline_service.py`

If either import fails (e.g. a missing dependency or syntax error in the route file), the router is silently not mounted and all endpoints return 404 with no log message visible to the frontend. This is a dangerous pattern — the container appears healthy but endpoints are missing.

#### `scripts/hor-integration-test/scenarios/` (all 9 files)

Files: `s1_crew_submit.py`, `s2_weekly_compliance.py`, `s3_weekly_signoff.py`, `s4_hod_countersign.py`, `s5_captain_sign_all.py`, `s6_fleet_inspect.py`, `s7_violation_notification.py`, `s8_crew_undo.py`, `s9_correction_flow.py`

The hardening removed all unconditional `check.ok()` calls and replaced them with checks that require:
- A specific HTTP status code
- A specific JSON field with a specific value (`check.expect_field`)

Pre-hardening example of a fake pass: `checks.append(check.ok("duplicate signoff idempotent"))` — always passed, proved nothing.
Post-hardening: a 409 response is a FAIL. A pass requires `success: true` in the JSON body.

Result after hardening: 9/9 scenarios pass, 92/92 checks pass, all backed by evidence.

#### `scripts/hor-integration-test/check.py`
**Purpose:** Assertion helper library used by all test scenarios.
**Status:** Edited

## 5. ISSUES, PROBLEMS, LIES, AND WHAT WENT WRONG

### 5a. False claims that were made and caught

**Session-opening claim: "9/9 scenarios passing."**
This was a lie by the prior session. A closer look showed many checks were unconditional `ok()` calls — they passed whether the API returned 200 or 500. A "409 duplicate error" was being reported as a pass. The tests proved nothing. Took a full audit + rebuild of all 9 scenario files to get real evidence.

**"The endpoints don't exist" (FRONTEND02's claim).**
Both `my-week` and `department-status` existed and were registered. The real cause: `NEXT_PUBLIC_API_URL` was not set to the local backend URL, so every request went to staging which lacked the latest code. No missing code — wrong environment variable.

**"Role enforcement will 403 HOD/master sign from wrong JWT."**
This engineer warned FRONTEND02 that using `crewPage` JWT for HOD/master sign steps would 403 after role enforcement was deployed. Upon reading the actual handler code: there is NO role enforcement in `sign_monthly_signoff`. It only checks sequential workflow state (crew → hod → master by `status`), not the caller's role. The warning was premature. Corrected immediately after reading the code.

The `/v1/hours-of-rest/fleet-compliance` endpoint is referenced in a comment but never fetched. The fleet view is completely non-functional — just a loading skeleton. No one fixed this.

**`signoffs/page.tsx` bypasses the proxy.** `apps/web/src/app/hours-of-rest/signoffs/page.tsx` line 105 fetches directly from `NEXT_PUBLIC_API_URL` instead of going through the Next.js `/api/v1/hours-of-rest/signoffs` proxy. **BUG-HOR-1 (schema):** FRONTEND02's test suite detects a wrong `user_id` in a record via `console.warn` but does not fail the test. The underlying schema issue (profile ID vs auth user ID mismatch) is a known but unresolved data integrity gap.

**Existing rows have `work_periods = null`.** All records created before the migration have no work_periods data. The slider will show empty on those days even though rest_periods data exists. A backfill is needed if historical data needs to be displayed correctly: compute `work_periods` as complement of `rest_periods` for all existing rows.

## 6. RESPONSE SHAPE CONTRACTS (for next worker)

### POST /v1/hours-of-rest/upsert

Input:
```json
{
  "record_date": "YYYY-MM-DD",
  "work_periods": [{"start": "09:00", "end": "22:00"}]
}
```
- `work_periods` can be empty array `[]` — means full rest day (24h rest)
- Backend derives `rest_periods`, stores both, calculates compliance

Output (standard envelope):
```json
{
  "success": true,
  "data": {
    "record": {"id": "uuid", "work_periods": [...], "rest_periods": [...], "total_rest_hours": 14.0, ...},
    "compliance": {"is_daily_compliant": true, "meets_mlc_minimum": true, ...}
  }
}
```

### GET /v1/hours-of-rest/my-week?week_start=YYYY-MM-DD

Output (FLAT — no envelope wrapper, not `{success, data}`):
```json
{
  "status": "success",
  "week_start": "YYYY-MM-DD",
  "week_end": "YYYY-MM-DD",
  "days": [
    {
      "record_date": "YYYY-MM-DD",
      "submitted": true,
      "work_periods": [{"start": "09:00", "end": "22:00", "hours": 13.0}],
      "rest_periods": [{"start": "22:00", "end": "09:00", "hours": 11.0}],
      "total_rest_hours": 11.0,
      "total_work_hours": 13.0,
      "is_daily_compliant": true,
      "warnings": []
    }
  ],
  "compliance": {
    "total_rest_hours": 77.0,
    "total_work_hours": 91.0,

    "days_compliant": 7,
```
